Fit Specification: Parsing and Output

 

When you run Fit, you give it an HTML file.  Fit parses that file, figures out where the tables are, interprets the tables according to fixtures, and outputs the results.  This portion of the Fit specification describes how Fit parses and outputs HTML documents.  First, we’ll talk about HTML document structure.  Then we’ll talk about individual table cells.

 

Fit only parses the tables in the HTML.  Within the tables, only the table cells are used.  Colspan and rowspan are ignored, but jagged tables (tables with a varying number of cells in each row) are okay.

In the preceding section, we talked about how Fit parses HTML documents.  Now we’ll talk about how Fit parses individual table cells.

 

Fit processes the data in each table cell.  It converts HTML into a plain string by performing several actions:

 

·        Stripping leading and trailing whitespace

·        Combining whitespace into a single space

·        Converting HTML entities into characters

·        Converting line break tags (“<br>”) into line feeds (“\n”).

Fit has the ability to modify cells before outputting them.  This is called “annotation.”  Fixtures annotate tables as they interpret them.  The standard annotations involve checking tables.  They have standard meanings:

 

·        none: Fit used the cell but didn’t check its value

·        right: Fit checked the cell against the software and the software worked properly

·        wrong: Fit checked the cell against the software and the software didn’t work properly

·        error: Fit checked the cell against the software and the software generated a run-time error (e.g., an exception)

·        ignore: Fit ignored the cell entirely

·        unchecked: The value in the cell wasn’t checked but the software provided this answer anyway (typically used when the cell is blank).

 

(For all of the annotation examples, the cell being modified starts out as “<td>Text</td>”.)
